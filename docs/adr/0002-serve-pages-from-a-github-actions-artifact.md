# Serve GitHub Pages from a GitHub Actions artifact, site/ instead of docs/

ADR-0001 put the deployed site at `docs/` on `main`, served via GitHub Pages' "deploy from a branch" mode (whose only folder choices are `/` or `/docs`). That collided with this repo's own engineering docs: the agent-skills toolkit in use here (`domain-modeling`, `code-review`, `improve-codebase-architecture`) hardcodes `docs/adr/` and `docs/agents/` as *its* default location, not as a configurable pointer — so `docs/` couldn't hold both the live website and the project's ADRs/agent config without an ongoing collision.

Chose to move the site to `site/` and switch GitHub Pages to "deploy from a GitHub Actions artifact" (`actions/upload-pages-artifact` + `actions/deploy-pages`) instead of a branch/folder deploy. That mode accepts any folder as the build output, so `site/` — or any name — works, freeing `docs/` for its toolkit-expected meaning. Trade-off: the weekly refresh Action (§5, not yet built) now needs the two extra Pages-deploy steps and `pages: write` / `id-token: write` permissions, rather than a plain `git commit && git push`. GitHub Pages itself isn't configured yet in this repo (checked via the API, still 404), so this is the cheapest point to make the change — no live deployment to migrate.
